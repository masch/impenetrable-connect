# Proposal: Integrate Project Standards into AGENTS.md

## 1. Change Summary

**What**: Move the user's custom Project Standards from `.atl/skill-registry.md` directly into `AGENTS.md`, making it the canonical source of truth.

**Why**: The `skill-registry` skill unconditionally overwrites `.atl/skill-registry.md` on every run (Step 4A: "ALWAYS write `.atl/skill-registry.md` regardless of any SDD persistence mode"). This causes the user's carefully crafted standards to be lost each session. Integrating into `AGENTS.md` protects them — the skill reads `AGENTS.md` in Step 2 and preserves its contents in the "Project Conventions" section.

## 3. Implementation

### 3.1 New AGENTS.md Structure

```markdown
# Agent Hub

This project uses **Agent Teams Lite (ATL)** and **Spec-Driven Development (SDD)**.

## 🧠 Mission Control

All technical standards, code conventions, and agent skill mappings are centralized in this file.

> ⚠️ **Important**: Project standards live directly in this file to prevent overwrites. The `.atl/skill-registry.md` file is auto-generated and may be overwritten.

## 📐 Project Standards

[Entire content from .atl/skill-registry.md sections goes here]

## 🛠️ Operational Model

1. **Specs First**: All changes must be backed by an OpenSpec in `openspec/specs/`.
2. **Auto-Load Skills**: The registry ensures that the AI assistant follows project-specific patterns (NativeWind v4, Drizzle, Hono).
3. **Deployment**: Backend deployment workflow and secrets management are documented in [docs/BACKEND_DEPLOYMENT.md](docs/BACKEND_DEPLOYMENT.md).
4. **Consistency**: Technical rules are maintained in the Project Standards section above.

---

## Project Standards (Auto-Resolved)

_(This section is auto-populated by the skill-registry skill — do not edit manually)_

[User Skills table will appear here after skill-registry runs]
```

### 3.2 How the Skill-Registry Reads It

The `skill-registry` skill executes:

1. **Step 2 (Scan Project Conventions)**: Finds `AGENTS.md`, reads its full contents, and extracts all referenced paths into the "Project Conventions" table.
2. **Step 3 (Write Registry)**: Generates the registry markdown with the User Skills table and Compact Rules.
3. **Step 4A (Persist)**: Writes `.atl/skill-registry.md` — but now it will be a minimal registry file without the Project Standards (since those live in AGENTS.md).

**Result**: The user's standards are preserved in AGENTS.md and discovered by the skill as a convention file. The generated `.atl/skill-registry.md` becomes a thin registry (skills + compact rules only).

### 3.3 Handling `.atl/skill-registry.md`

**Option**: Keep it as a thin registry (recommended)

- Remove the Project Standards sections from `.atl/skill-registry.md`
- Let the skill-registry generate only the skill registry portion
- The file becomes: User Skills + Compact Rules + Project Conventions reference to AGENTS.md
- Update AGENTS.md to remove the link to `.atl/skill-registry.md`

**Alternative**: Remove entirely

- Delete `.atl/skill-registry.md` from the repo
- The skill-registry will recreate it anyway (Step 4A is unconditional)
- This creates churn but keeps AGENTS.md as sole source

**Decision**: Keep as thin registry — less disruptive.

## 4. Risks

| Risk                                                        | Severity | Mitigation                                              |
| ----------------------------------------------------------- | -------- | ------------------------------------------------------- |
| Skill still overwrites `.atl/skill-registry.md`             | Low      | Standards now in AGENTS.md; thin registry is acceptable |
| Convention file reference in registry table is insufficient | Low      | Step 2 reads full AGENTS.md content, not just path      |
| Future skill-registry updates change behavior               | Low      | Monitor on skill updates; documentation in AGENTS.md    |
| Merge conflicts if user edits AGENTS.md manually            | Medium   | Standards are single source; edits should go there      |

### Edge Cases

1. **User edits `.atl/skill-registry.md` directly**: They should edit AGENTS.md instead. Add a note in `.atl/skill-registry.md` pointing to AGENTS.md.

2. **Running skill-registry manually**: It will regenerate the thin registry. Standards in AGENTS.md are untouched.

3. **SDD sessions overwrite everything**: Even in SDD, AGENTS.md is a convention file the skill reads, not a file it overwrites. Standards survive.
